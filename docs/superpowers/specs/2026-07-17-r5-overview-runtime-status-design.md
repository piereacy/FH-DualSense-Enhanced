# Enhanced R5 总览运行状态设计

## 目标

让 GUI 总览中的 DualSense、Forza 遥测、当前配置和更新四张状态卡成为可信的实时摘要，而不是停留在构建时默认文案。状态必须能帮助用户区分正常等待、正在工作、连接中断和明确故障，同时保持现有单一青绿色界面、快捷入口和后端输出边界不变。

本设计属于 Enhanced R5。当前公开稳定版仍是 Enhanced R4，直到后续完成构建、验证和发布。本设计不包含 FH6 中文文字与英文语音资源交换，该功能作为独立的游戏文件管理子系统另行设计。

## 当前问题

- `OverviewTab.refresh()` 已经读取四类状态，但初始化后没有立即执行，现有每秒 `_tick_status()` 也没有调用它，因此总览长期显示“等待中”“-”和“更新状态：空闲”。
- 现有遥测判断依赖由运行循环临时写入的 `listener.lost` 属性。监听器本身没有稳定状态模型，也无法可靠区分等待首包、正在接收和收到过数据后的中断。
- UDP 绑定或后端启动失败只临时写入顶部状态条，下一次状态刷新可能重新覆盖为普通等待状态。
- 更新服务、Profile 和 DualSense 已有可读取状态，但总览没有统一、可单元测试的展示映射。

## 设计边界

- 保留每秒由 Tk 主线程拉取状态的模式，不引入跨线程事件总线。
- 后端线程不得直接访问 Tk widget。
- 不改变 HID report、DualSense 重连、UDP drain、触觉静音、更新下载或 Profile 持久化行为。
- 四张卡继续是只读摘要。操作入口仍位于快捷入口、配置文件页和系统与更新页。
- 不恢复 GUI 变体，不改变现有卡片布局和主题。

## 遥测状态模型

`UDPListener` 增加线程安全的不可变遥测快照。快照至少包含：

- 有效包数量。
- 最后一个有效包的 `time.monotonic()` 时间戳。
- 最后一个有效包的来源 IP。
- 当前监听端口。
- 派生阶段：等待首包、正在接收或接收中断。

快照只把长度严格等于 `PACKET_SIZE = 324` 的数据包计为有效遥测。长度错误的数据包继续沿用现有“每种长度只警告一次”的行为，但不得把状态切换为正在接收。

阶段按读取快照时的单调时间派生：

- 监听成功但有效包数量为零：等待首包。
- 最后一个有效包距今不超过 1 秒：正在接收。
- 曾经收到有效包，但超过 1 秒没有新有效包：接收中断。

1 秒阈值与现有触觉静音时机一致。现有 5 秒无包日志警告和 `telemetry_lost_exit_s` 退出规则保持不变。状态锁只保护少量字段复制与更新，不包住阻塞式 `recvfrom()` 或 drain 循环。

UDP 绑定失败发生在监听器创建完成之前，因此由 `TriggerGUI` 保存独立的后端启动错误状态。该状态同时供顶部状态条和总览使用，直到后端成功重建或应用退出。

## 展示映射

新增不依赖 Tk 的纯展示映射，将运行时对象转换为卡片的短主文案和提示文案。每个映射失败时只影响对应卡片，不阻断其他卡片刷新。

### DualSense

- 原生后端未连接：主文案为“等待手柄”，提示 USB/Bluetooth 以及自动重连状态。
- 原生后端已连接：主文案为“已连接”，提示 `USB` 或 `Bluetooth`；保持连接模式继续如实显示。
- DSX 后端：主文案为“DSX 已启用”，提示目标地址和无 ACK 的 UDP 语义，不把 socket 已打开描述成物理手柄已连接。
- 后端启动失败：显示短错误状态，详细原因放入提示或日志。

### Forza 遥测

- 监听成功但尚无有效包：主文案为“等待遥测数据”，提示已监听的 UDP 端口。
- 正在接收：主文案为“正在接收”，提示本地端口和最后来源 IP。
- 接收中断：主文案为“遥测已中断”，提示最后接收距今时间和本地端口。
- UDP 绑定失败：主文案为“监听失败”，提示不可用的端口，不显示普通等待状态。

### 当前配置

- 正常时显示持久化存储中的活动名称，包括 `Default` 和命名 Profile。
- 提示继续说明“更改即时保存”。
- 读取失败时显示“读取失败”，不退回无含义的 `-`。

### 更新

- 区分等待自动检查、自动检查已关闭、检查中、已是最新版、发现新版本、下载进度、校验中、等待安装、安装重启和检查失败。
- 下载主文案包含整数百分比，提示保留目标 Release 或运行环境信息。
- 支持时提示 Windows 独立 EXE 和最近检查时间。
- 源码、Linux 与 ZUV 明确显示当前运行环境不支持内置 EXE 替换。
- 继续复用 `UpdateService.snapshot()`、系统页按钮逻辑和导航白点判断。

原始异常文本不得直接放入大字号主文案。展示层生成固定的短错误标题，并对提示长度进行限制，完整异常仍写入日志。

## 刷新流程

1. `OverviewTab` 完成控件构建和回调注册后立即执行一次刷新。
2. `TriggerGUI._start_backend()` 成功或失败后请求一次主线程刷新。
3. 现有 `_tick_status()` 每秒依次刷新顶部状态条、总览和更新导航白点。
4. Profile 创建、应用、重命名、恢复默认或退出另存后，现有设置刷新路径继续触发总览更新；每秒轮询负责兜底。
5. teardown 开始后停止后续调度，不从已关闭的后端读取状态。

每张卡独立捕获状态读取错误。相同错误不应每秒重复写日志，避免故障时产生日志风暴。

## Enhanced R5 版本边界

- `src/pyproject.toml` 的内部 PEP 440 版本改为整数 `5`，公开开发身份由现有映射显示为 `R5`。
- `src/uv.lock`、打包版本资源、规范 EXE 名称和测试期望同步为 `FH-DualSense-Enhanced-R5.exe`。
- 保留整数 `R<n>` tag 与资产契约，不引入 `R4.1`。这样已经发布的 R4 更新器可以把 R5 判断为更高稳定版本。
- Release workflow 的完整中文与英文正文改为 Enhanced R5，并说明实时总览状态；本任务不创建 tag、不推送 Release，也不修改已发布的 R1 至 R4 资产。
- 根 README 与中日文 README 不增加该界面内部细节。架构、决策和项目状态文档按各自触发规则同步，明确 R5 已实现但尚未发布。

## 测试与验收

### 自动测试

- 纯展示映射覆盖四张卡的全部正常、等待、中断、不支持和错误阶段。
- UDP 快照覆盖等待首包、有效 324 字节包、错误长度、1 秒中断、恢复接收和来源 IP 更新。
- GUI 契约测试确认总览在初始化后刷新，并由 `_tick_status()` 每秒刷新。
- 后端启动失败测试确认顶部状态条和总览不会在下一次 tick 被覆盖为普通等待。
- 更新测试覆盖等待自动检查、关闭自动检查、下载百分比和错误短文案。
- 版本回归测试确认 `current_version=4` 能识别规范的 R5 Release 和 R5 EXE/sidecar。
- 非英语语言目录覆盖新增的固定文案。

### 执行验证

- 运行最接近的 GUI、UDP、Profile、更新和发布契约定向测试。
- 运行 `uv run --project src pytest -q`。
- 运行 `python -m compileall -q src/modules src/lang` 和 `git diff --check`。
- 启动源码 GUI，目视确认四张卡在现有布局中不会因最长固定文案裁切，并验证等待状态和可模拟状态变化。
- 没有真实 FH6 或 DualSense 参与时，真实遥测、USB 和 Bluetooth 验证明确记录为“未执行”，不得推断为已通过。

## 非目标

- 不新增数据包速率图表、历史时间线或可点击状态卡。
- 不在总览显示原始异常堆栈、完整手柄序列号或更新 Release 正文。
- 不改变 DSX 无 ACK 的能力边界。
- 不实现自动更新 24 小时节流、PE 版本解析或代码签名。
- 不发布 Enhanced R5，也不修改已发布 Enhanced R4。
